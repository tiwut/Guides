Project Midori: The Non-Windows Dream

While Longhorn and 10X tried to fix Windows, Project "Midori" tried to replace it entirely. Discover Microsoft's secret research operating system built from scratch using managed code.

Section 1: What was Project Midori?

üåø
NOT WINDOWS: Unlike every other project in this archive, Midori was not based on the Windows NT kernel. It had zero code in common with Windows.

Incubated deeply within Microsoft Research around 2003 (and active until roughly 2015), Midori was an answer to a specific question: "If we could start over today, without worrying about backward compatibility, how would we build an OS?"

The answer was Safety. Most OS crashes and security exploits (buffer overflows) come from memory management errors in C and C++. Midori aimed to eliminate these bugs at the architectural level.

Section 2: System M# (The Language)

‚å®Ô∏è

You cannot write a safe OS in C++. So, the Midori team created a new programming language, often referred to as M# (M-Sharp) or System C#.

  • Type Safety: The language prevented the programmer from making memory mistakes. A program literally could not access memory it didn't own.
  • No Garbage Collection Pauses: Usually, managed languages (like Java or C#) pause the system to clean up memory. Midori used a unique ownership model (similar to how Rust works today) to ensure high performance without pauses.
  • Singularity: Midori was the commercial evolution of an earlier academic project called "Singularity," which proved that an OS could run efficiently in managed code.

Section 3: The Asynchronous Architecture

‚ö°

Midori was designed for the cloud era, where one application might span across multiple servers.

"Async Everything": In Windows, if an app waits for a file to load, it "blocks" a thread. In Midori, the entire OS was asynchronous. Nothing ever blocked. This made the system incredibly lightweight and responsive.

Isolation: Programs didn't share memory. They communicated by sending small messages to each other. If one part of the system crashed (e.g., the network driver), it was just a "process" that could be restarted instantly without a Blue Screen of Death.

Section 4: Why it Failed & The Legacy

üß¨

Midori was technically superior to Windows in almost every way. So why aren't we using it?

The "Eco-System" Problem: Moving to Midori would have meant abandoning millions of existing Windows applications. Microsoft concluded that the cost of rewriting the world's software was too high.

However, Midori was stripped for parts, and its DNA is everywhere in modern Microsoft tech:

Midori Tech Where it lives today
M# / System C# The performance improvements for managed code were ported into .NET Core and C#. Features like Span<T> (high-performance memory slicing) came directly from Midori.
Async/Await The asynchronous programming model in C# (and subsequently JavaScript/TypeScript) was heavily influenced by the Midori team's work.
Rust Influence Many of the concepts Midori pioneered (memory safety without garbage collection) are now mainstream in the Rust programming language, which Microsoft is now adopting for parts of the Windows Kernel.
Conclusion: Midori was the "Formula 1 car" of operating systems. It was too specialized for the street, but the technology developed for it improved the engine of the car we all drive (Windows/.NET).
