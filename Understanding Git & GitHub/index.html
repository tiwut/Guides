Understanding Git & GitHub

A complete guide to version control, repository management, and collaborative development workflow.

Introduction

Git is a distributed version control system that tracks changes in your source code during software development. It allows multiple developers to work together without overwriting each other's changes.

GitHub is a cloud-based hosting service that lets you manage Git repositories. It adds collaborative features like Pull Requests, Issue Tracking, and CI/CD pipelines to the core Git functionality.

Core Concepts

  • Repository (Repo): The folder containing your project files and the entire revision history.
  • Commit: A snapshot of your changes at a specific point in time. Think of it as a "save point."
  • Branch: A parallel version of your repository. Use branches to develop features in isolation before merging them into the main code.
  • Merge: The process of taking changes from one branch (e.g., a feature branch) and combining them into another (e.g., main).
  • Remote: The version of your repository hosted on the internet (e.g., on GitHub).
Concept: Git is local (on your machine), while GitHub is remote (on the web). You "push" local changes to the remote and "pull" remote changes to your local machine.

Basic Commands

Here are the essential commands to configure Git and start tracking a project:


# 1. Configure your identity
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"

# 2. Initialize a new repository
$ git init

# 3. Stage changes (prepare for commit)
$ git add .

# 4. Save changes (create a commit)
$ git commit -m "Initial commit"

# 5. Connect to GitHub and upload
$ git branch -M main
$ git remote add origin https://github.com/username/repo.git
$ git push -u origin main

Use git status at any time to see which files have changed.

Workflow & Best Practices

To maintain a clean history and avoid conflicts, follow these guidelines:

Security Warning: Never commit sensitive data like API keys, passwords, or environment files (.env). Use a .gitignore file to exclude them.
  • Commit Often: Make small, logical commits rather than one massive update.
  • Clear Messages: Write imperative commit messages (e.g., "Fix login bug" instead of "Fixed bug").
  • Use Branches: Never push directly to the main branch. Create a new branch for every feature or bug fix.
  • Pull Requests: Use GitHub Pull Requests to review code before merging it into the main branch.
